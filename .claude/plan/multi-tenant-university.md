## 1. Requirement (restated)

The system must support **multi-tenancy with an organizational hierarchy** so it can be
deployed at two scales:

1. **University-wide deployment** ‚ÄĒ a *super admin at the university level* sees a
   roll-up across **colleges**: running exams across all colleges, live alerts across
   colleges, per-college health/KPIs, drill-down into any college's exam.
2. **Single-institution deployment** ‚ÄĒ a lone college/school installs the system on its
   own. There is no university parent; the existing single-tenant experience must keep
   working unchanged (no empty "university" chrome, no extra setup friction).

The same binary/schema must serve both; the difference is purely *data shape* (is there a
parent org with children?) ‚ÄĒ not a separate build or feature flag the operator must set.

**Consequence:** Multi-tenancy is not just a "new dashboard" ‚ÄĒ the data layer has *no
tenant boundary at all*. The work is therefore (a) introduce an org hierarchy, (b) make
"access scope = the subtree of institutions rooted at the user's institution" the single
authorization primitive, and (c) build the university roll-up on top of that primitive.
Item (b) is simultaneously the feature enabler **and** a latent security hardening.

## 3. Technical Solution (synthesized)

### 3.1 Core model ‚ÄĒ hierarchical `Institution` (self-referential)

Reuse the existing `Institution` table instead of inventing a separate `Organization`
table. Add:

- `parent_id: UUID | None ‚Üí institutions.id` (self-FK, `ondelete=RESTRICT`).
- `type: str` ‚ÄĒ one of `university` | `college` | `school` | `standalone`
  (default `standalone`).

Semantics:
- **University deployment**: one root institution `type='university'`, `parent_id=NULL`;
  each college is an institution with `parent_id = university.id`, `type='college'`.
- **Single deployment**: one institution, `parent_id=NULL`, `type ‚ąą {college, school,
  standalone}`. No children.

A university super admin is simply a `super_admin` **whose `institution_id` is the root
(parent) institution**. A college super admin is a `super_admin` whose institution is a
leaf college. *Role stays identical; scope differs by where the user sits in the tree.*
This avoids adding a new role and keeps the role-merge migration coherent.

**Depth is capped at 2 (university ‚Üí college)** per resolved Q3. `parent_id` could support
deeper trees, but recursion/UI assume exactly two levels for now; a future ministry tier is
a non-breaking extension.

### 3.2 The one scoping primitive: `accessible_institution_ids(user)`

Add a helper (e.g. `src/thaqib/core/scoping.py`) that returns the set of institution IDs a
user may read/act on:

```

    # subtree rooted at the user's own institution
    root = user.institution_id
    return {root} | descendant_ids(db, root)   # recursive CTE or iterative BFS

```

- Leaf college / single deployment ‚Üí `{own_id}` (one element) ‚Üí behaves exactly like
  today's single tenant.
- University root ‚Üí `{root} ‚ą™ all colleges` ‚Üí cross-college visibility, automatically.

Every list/read/aggregate query filters on `institution_id IN accessible_set`. This is the
seam that makes both deployment shapes fall out of one code path.

### 3.3 Denormalize `institution_id` onto `ExamSession`

Add `ExamSession.institution_id` (FK, NOT NULL going forward) set to the owning college at
creation time (derived from the halls' institution, validated to be single-college). This
makes the cross-college roll-up a simple indexed `GROUP BY institution_id` instead of a
3-table join through `exam_session_halls`. Backfill in the migration from existing halls.

> Decision: keep one exam = one college (a session's halls must all belong to the same
> institution). Validated at create/update. This keeps alerts/events unambiguously
> attributable to a college for the roll-up. (Open question Q1 below if cross-college
> exams are ever needed.)

### 3.4 University aggregation API (new router `overview.py`)

New read-only endpoints, all scoped through `accessible_institution_ids`:

- `GET /api/overview/colleges` ‚Üí per-college cards: name, # running exams, # halls ready,
  # active alerts (pending+claimed), # invigilators online. Only meaningful when the
  caller's subtree has >1 institution; for a single college it returns the one card.
- `GET /api/overview/exams?status=active` ‚Üí running exams **across** the subtree, each
  tagged with `college_name`, hall count, live alert count.
- `GET /api/overview/alerts` ‚Üí merged live alert feed across the subtree (DB-backed
  `Alert` rows joined to exam‚Üíinstitution), filterable by `college_id`.
- `GET /api/overview/summary` ‚Üí top-line KPIs for the header (total running exams, total
  active alerts, colleges-with-activity), plus `is_multi_college: bool` so the frontend
  can decide which dashboard to render.

**`is_multi_college` is derived, not a stored flag (resolved Q4):**
`is_multi_college = (user.institution.type == 'university')`. Keying off the *type* (the
user's own node being a container) rather than `len(subtree) > 1` means a freshly-installed
university with **zero colleges yet** still renders the university shell with an
"add college" call-to-action, instead of falling back to the single-college view. A
`college`/`school`/`standalone` institution is always single-tenant. No separate
"multi_tenant" boolean is persisted ‚ÄĒ the institution `type` is the single source of truth.

### 3.5 Retrofit existing endpoints with the scoping primitive

This is the security-relevant half. Replace "optional `institution_id` query param" and
"no filter at all" with enforced subtree filtering:

- `halls`, `users`, `devices`, `institutions` list/read ‚Üí filter to
  `accessible_institution_ids`; reject cross-tenant IDs with 404 (not 403, to avoid
  leaking existence).
- `exams` list/read/report ‚Üí filter `ExamSession.institution_id IN accessible_set`
  (super_admin sees the subtree; admins remain further narrowed by `ExamAdminAssignment`).
- `stream` live feeds (`/monitoring`, `/status`, `/alerts`) ‚Üí filter the in-memory
  manager's hall/camera set to halls whose `institution_id ‚ąą accessible_set`. (The manager
  already keys cameras by hall; add an institution filter pass before serialization.)
- `institutions` create ‚Üí a university super admin may create child colleges
  (`parent_id` = their root); a single-tenant super admin keeps creating peers only if
  explicitly allowed (default: disabled once installed). 

### 3.6 Frontend ‚ÄĒ university dashboard + graceful degradation

- Fetch `GET /api/overview/summary` once on load. If `is_multi_college` is **false**,
  render the **existing** `DashboardPage` unchanged (zero visual change for single
  colleges).
- If **true**, render a new **`UniversityDashboardPage`**:
  - Header KPIs (running exams, active alerts, active colleges).
  - **College grid** (cards from `/overview/colleges`) ‚ÄĒ click a card to scope down.
  - Cross-college **live alerts** panel (`/overview/alerts`) with a college filter chip
    row; clicking an alert opens the existing `CameraModal`/alert detail.
  - Cross-college **running exams** list (`/overview/exams`) ‚Üí drill into a college's
    existing dashboard view scoped to that `college_id`.
- Add a **college switcher** in the nav (replaces the flat hall selector at the top
  level): "All colleges" ‚Üí per-college ‚Üí per-hall.
- Reuse existing `CasesTab` / `CamerasTab` / `CameraModal`; the university view is mostly a
  new shell + an aggregation data source, not a rewrite of the alert/camera widgets.

## 6. Risks and Mitigation

| Risk | Mitigation |
|------|------------|
| Retrofitting scope filters is a latent **security boundary change** ‚ÄĒ easy to miss an endpoint and leak cross-tenant data. | Centralize in one `get_scope` dependency; add `tests/test_tenant_isolation.py` that asserts tenant A's token returns 404/empty for every list+detail endpoint on tenant B's resources. Treat any unscoped query as a review blocker. |
| Recursive `parent_id` query differs between **PostgreSQL (prod)** and **SQLite (tests)**. | `descendant_ids` uses a `WITH RECURSIVE` CTE on PG and an iterative BFS fallback detected via dialect; cover both in `test_scoping.py`. Cap depth (e.g. 2: university‚Üícollege) to avoid runaway recursion. |
| **Backfill correctness** for `exam_sessions.institution_id` when a legacy session has halls from >1 institution (shouldn't exist, but data may be dirty). | Migration logs/skips ambiguous rows, assigns the most common hall institution, and the validation rule prevents new violations. Make column nullable in the migration, backfill, then enforce NOT NULL in a follow-up once verified. |
| **Single-tenant regression** ‚ÄĒ existing colleges must see zero change. | `is_multi_college` gates all new UI; when the subtree has one node the new filters reduce to `institution_id == own` (current behavior). Keep existing `DashboardPage` as the single-college renderer. |
| Live `/stream` feeds are **in-memory, not DB-scoped** ‚ÄĒ filtering them is the trickiest retrofit. | Add an institution filter pass over the manager's hall-keyed camera map before serialization, using a hall‚Üíinstitution lookup cached per request. Verify with a 2-college fixture that each super admin sees only its halls. |
| University super admin **acting** (not just viewing) across colleges could violate college autonomy. | Keep writes (create exam, assign invigilator, claim alert) bound to the owning college's admins via existing `ExamAdminAssignment`; the university role is **observe + drill-down** by default. Revisit per Q2. |

## 7. Resolved Decisions (was Open Questions)

- **Q1 ‚ÄĒ RESOLVED: one exam = one college.** A single exam cannot span halls across
  colleges. `ExamSession.institution_id` stays a single FK; create/update validates that all
  selected halls belong to the same institution. No M:N needed.
- **Q2 ‚ÄĒ RESOLVED: observe-only.** The university super admin **observes and drills down**;
  it does not act across colleges. All writes (create exam, assign invigilator, claim/resolve
  alert) remain bound to the owning college's admins via `ExamAdminAssignment`. The
  `/api/overview/*` endpoints are strictly read-only.
- **Q3 ‚ÄĒ RESOLVED: 2 levels (university ‚Üí college).** Cap recursion/UI at depth 2.
  `parent_id` keeps a deeper tree possible later without a migration, but scoping and labels
  assume exactly two tiers now.
- **Q4 ‚ÄĒ RESOLVED: setup picks the institution type; tenancy is derived from it.**
  The setup wizard adds an institution-**type** selector (`University` | `College` |
  `School` | `Standalone`). Choosing `University` *is* the multi-tenant decision ‚ÄĒ the root
  becomes `type='university'` and the wizard offers an optional "add colleges" step (children
  with `parent_id = root.id`); colleges can also be added later from the dashboard. Any other
  type is single-tenant. There is **no separate "is this multi-tenant?" question and no stored
  flag** ‚ÄĒ `is_multi_college` is computed as `user.institution.type == 'university'`, so even a
  university with zero colleges yet renders the university shell.
